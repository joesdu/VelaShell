# SFTP 双栏与 WinSCP 差距分析

> 编写日期：2026-07-22　　基准代码：`tmds-ssh` 分支
>
> 用途：**决策清单**。逐项列出 VelaShell 双栏 SFTP 与 WinSCP 的差距，供后续挑选要实现的功能。
> 每一项都标注了当前状态与实现代价，未核实的一律标注"未核实"，不做推测。

---

## 一、先看结论

VelaShell 目前是一个**做得相当扎实的单向文件传输器**，但还不是一个**双栏文件管理器**。

传输管道本身的质量明显高于同类小工具：递归上传/下载规划、四种冲突策略、断点续传（含起点核实）、
限速、并发、进度浮窗、传输日志、取消语义、`LocalPathSafety` 路径穿越校验。
远端栏的表格（7 列、可排序、可隐藏、可拖宽、双击自适应）也超出预期。

差距集中在三个层面，**处理方式完全不同，不要混在一起排期**：

| 类别 | 含义 | 用户感知 | 建议 |
|---|---|---|---|
| **A. 接线债** | 代码写好了，但没接到双栏这条路 | **像 bug** | 优先修，代价极低 |
| **B. 能力缺失** | 确实没做，但架构支持 | 功能少 | 按需求排期 |
| **C. 架构级缺失** | 连接口层都没预留 | 与 WinSCP 的本质差距 | 需要专门设计 |

## 二、A 类：接线债（建议优先处理）

这些**不是"没做"，是"做了但没接上"**。用户会当成 bug 报，修复成本却很低。

| # | 问题 | 位置 | 现象 |
|---|---|---|---|
| A1 | **外部编辑器在双栏里永远报"未配置"** | `SftpDocumentViewModel.cs:33-40` 未赋值 `GetDefaultEditorPath` | 用户在设置里配了编辑器也用不了。终端侧宿主有接（`MainWindowViewModel.cs:846`），双栏漏了 |
| A2 | **本地栏表头无法排序** | `LocalFilePaneView.axaml:128-138` 表头是纯 `TextBlock` | `SortCommand`（`LocalFilePaneViewModel.cs:49`）与 `ToggleSort:499` 完整可用，就是没绑 |
| A3 | **传输浮窗无法单条取消 / 重试 / 清空** | `FileTransferView.axaml` 只绑了 `CancelAllCommand`、`HidePanelCommand` | 三个命令（`FileTransferViewModel.cs:106-112`）没有对应控件。**且 `RetryTransfer:419` 本身只改状态不重发**，接上 UI 也是空动作 |
| A4 | **`TransferManager` 整套队列零调用点** | `Core/Sftp/TransferManager.cs` | 排队 + 并发 + 取消都实现了、DI 也注册了，但生产代码从不调用 `QueueTransferAsync`。浏览器自己用 `SemaphoreSlim` 跑 |
| A5 | **远端栏"上级目录"没有按钮** | `GoUpCommand`（`FileBrowserViewModel.cs:648`）无 axaml 绑定 | 只能双击 `..` 行。本地栏有按钮 |
| A6 | **双栏关掉远端栏后无法恢复** | `ToggleVisibilityCommand` → `IsVisible=false` | 该按钮是给终端宿主设计的（那边有重开路径），双栏里点了就只剩空白 |
| A7 | **双栏的传输设置是构造时快照** | `SftpDocumentViewModel.cs:36-37` | 改了设置对已打开的 SFTP 文档不生效；`OnSettingsSaved` 只遍历终端侧缓存 |
| A8 | **双栏不读/不写列显示与隐藏文件设置** | `SftpDocumentViewModel.cs:33-40` | 终端侧接了 `ShowHiddenFiles`、`ColumnVisibilityToggled`，双栏没接 |
| A9 | **`DownloadItemCommand` 无绑定** | `FileBrowserViewModel.cs:670` | 连带 `PickSavePathForDownload`（"下载单文件时另存为"）在 UI 上不可达 |
| A10 | **续传下载不套限速** | `SftpService.cs` 续传分支未包 `ThrottledStream` | 断点续传时带宽限制静默失效 |
| A11 | **传输日志把 Copy 记成 DOWNLOAD** | `TransferLogService.cs:40` 三元只区分 Upload/其它 | 日志类型不准 |
| A12 | **OS 拖文件到本地栏只刷新、不复制** | `LocalFilePaneView.axaml.cs:234-243` | 拖进去什么也没发生 |

> **重复定义**：`AppSettings.AutoResume`（:623）与 `ResumeEnabled`（:668）语义重叠，只有后者被消费。
> `TransferMaxRetries`（:671）、`AutoCleanTempFiles`（:674）代码注释自认"规划中，无运行时消费者"。

## 三、B 类：能力缺失（按 WinSCP 功能域对照）

### 3.1 浏览与导航

| WinSCP 能力 | VelaShell | 说明 |
|---|---|---|
| 可编辑路径输入框 | ❌ 无 | 两侧都只有面包屑，无法直接粘贴路径跳转。**高频刚需** |
| 前进 / 后退历史 | ❌ 无 | 两个 VM 都没有历史栈 |
| 书签 / 收藏夹 | ❌ 无 | WinSCP 用户依赖度很高 |
| 目录访问历史下拉 | ❌ 无 | |
| 面包屑 | ✅ 双侧已实现 | |
| 隐藏文件开关 | ⚠️ 仅远端 | 本地栏永远显示隐藏文件且无开关 |
| 本地盘符切换 | ✅ 已实现 | |

### 3.2 文件操作

| WinSCP 能力 | VelaShell | 说明 |
|---|---|---|
| 新建文件夹 / 文件 | ✅ 远端齐全 | 本地栏**只有新建文件夹，没有新建文件** |
| 重命名 | ✅ 双侧 | |
| 删除（递归 + 进度 + 取消） | ✅ 双侧 | 质量不错，远端还有加权进度 |
| 移动 / 复制（远端→远端） | ⚠️ 要手敲绝对路径 | 无目录选择器、无拖放移动 |
| chmod | ⚠️ 仅单文件 | 有 9 格 rwx 矩阵 + 八进制同步，但**无批量、无递归应用、无 setuid/sticky 位** |

| **符号链接创建 / 识别** | ❌ 无 | `RemoteFileInfo` 无符号链接字段，权限串首字符只有 `d`/`-`，链接被当成普通文件 |

| 本地文件属性 / 权限 | ❌ 无 | 本地栏右键只有 5 项 |

## 四、C 类：架构级缺失

这三项**连接口层都没有预留**（`ISftpService.cs` 全文 82 行无相关签名），需要专门设计。

### C1. 同步与目录比较（**与 WinSCP 最本质的差距**）

WinSCP 的三大同步能力，VelaShell **零行代码**：

- **目录比较**：并排标出两侧的"仅左有 / 仅右有 / 不同 / 相同"
- **同步（单向 / 双向 / 镜像）**：预览差异 → 确认 → 批量执行
- **保持远程目录最新**：监视本地改动自动上传

这是很多人选择 WinSCP 的**唯一理由**。要做需要新增：
比较结果模型、差异计算（按大小 + mtime，可选 checksum）、预览 UI、执行引擎（复用现有传输管道）。

### C2. 搜索能力

`ISftpService` 需要新增递归搜索 API。注意远端递归遍历的成本，
需要流式返回 + 可取消（现有 `ListDirectoryAsync` 是一次性返回 `List`，不适合直接递归）。

### C3. 终端联动

- 文件栏右键无"在此处打开终端"
- 终端当前目录无法同步到 SFTP 栏

SFTP 文档目前是纯双栏（`SftpDocumentView.axaml` 无终端控件）。
本项目**已有完整终端栈**，这一项的架构基础其实是现成的，主要是 UI 编排。

## 五、建议的优先级

纯粹从"投入产出比"排，供参考：

**第一梯队（低成本、高感知）**
1. A 类接线债（A1、A2、A3、A5、A6、A7、A8）——用户当成 bug 的那些
2. 键盘快捷键（F5 刷新 / F2 重命名 / Del 删除 / Enter 进入 / Ctrl+A 全选）
3. 可编辑路径输入框
4. 当前目录过滤框

**第二梯队（中等成本、明确需求）**
5. 通配符选择 / 反选
6. 传输队列可视化（A4 + 单条取消重试）
7. 列宽与分栏比例持久化
8. 本地栏补齐（排序、隐藏文件开关、新建文件）
9. 拖放到目标文件夹行

**第三梯队（高成本、差异化）**
10. **目录比较 + 同步**（C1）——决定"是否能替代 WinSCP"
11. 远端递归搜索（C2）
12. 终端联动（C3）
13. chown / 符号链接 / 时间戳修改
14. 传输后校验

## 六、附：2026-07-22 已完成的编辑器增强

内置编辑器（`RemoteFileEditorView`）原先**完全没有语法高亮**。已补：

- **文件类型自动识别**（`Services/Syntax/FileTypeDetector.cs`）：
  扩展名 → 特殊文件名（`Dockerfile`/`sshd_config`/`fstab`/`.bashrc`…）→ **shebang**。
  第三级对远端编辑尤其关键：服务器上大量可执行脚本没有扩展名，只有 `#!/bin/bash` 能说明它是什么。
- **补齐 AvaloniaEdit 缺失的语法定义**（`Syntax/*.xshd`）：
  内置 20 份定义里**恰恰没有** Shell、YAML、INI/conf、Dockerfile、Log ——
  而这五种正是运维日常编辑最多的。已自行编写。
- **主题跟随**（`SyntaxHighlightingService`）：
  AvaloniaEdit 内置定义是给浅色背景调的（关键字 `Blue`、标点 `Black`），
  在本应用的 Dracula 面 `#282A36` 上标点等于隐形。现按命名颜色整体重着色为
  Dracula（暗）/ Alucard（亮），未收录的角色做对比度兜底。

**已知限制**：编辑器打开后再切换主题，配色不会实时更新（重新打开即可）。
