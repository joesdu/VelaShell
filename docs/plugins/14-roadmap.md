## 风险登记

| 风险 | 影响 | 缓解 |
| --- | --- | --- |
| ALC 共享程序集上钻的类型同一性问题(多版本 StreamJsonRpc 混载) | M1 受阻 | M0 spike 专项覆盖;共享白名单机制尽早定稿 |
| 每插件一进程的内存开销引发用户抱怨 | 口碑 | 惰性激活 + 空闲回收(03 §7);管理页透明展示占用 |
| VelaUI 表达力不足,开发者抱怨"做不出想要的 UI" | 生态 | 三示例先行自验;白名单控件按需求单渐进扩容(apiLevel 内加法) |
| 音频后端选型踩坑(跨平台输出) | M7 延期 | C-6 先 spike;最坏降级为"仅 Windows 首发音频域",能力协商机制天然支持按平台宣告 |
| 权限弹窗打断感强,用户体验反弹 | 口碑 | 范围收窄 + 选择器免权限路径(07 §5)引导插件少要权限;详情页事前展示 |
| 单人带宽不足导致战线过长 | 全局 | 里程碑闸门制,每个里程碑结束都是可暂停的稳定点;M4 起示例即验收,避免"最后一刻集成" |

## 执行纪律

- 每个里程碑收尾:更新本目录对应文档(设计即文档,漂移即技术债);
  闸门未过不开下一里程碑。
- 所有新增公共面(权限 id、贡献点、RPC 方法)必须先改 03/06/07 文档
  再写代码。
- 与主干开发的关系:插件系统在长期特性分支 `feature/plugins` 推进,
  按里程碑合回 dev(M1 起宿主侧改动即影响主程序,须过全量既有测试)。
