# 12 · 安全与威胁模型

## 1. 信任模型(分层陈述,禁止夸大)

| 资产 | v1 保护强度 | 依据 |
| --- | --- | --- |
| SSH 凭据(密码/私钥/passphrase) | **强**:物理上只存在于宿主进程与 OS 凭据库,无任何 RPC 可读取 | D5 |
| 会话通道(SFTP/exec/终端) | **强**:仅经宿主能力 API 可达,每次调用过 Broker,可撤销、有审计 | 06 |
| 宿主 UI 与其它插件 | **强**:进程隔离 + UI 全部宿主渲染;插件间无通道 | D1/D6 |
| 插件私有数据 | **中**:目录按插件划分 + secrets 命名空间隔离;同一 OS 用户下无强制隔离 | — |
| 本地文件系统 | **弱(v1)**:经 API 的访问受 Broker;但插件进程可直接 syscall 绕过 | 见 §3 |
| 网络出口 | **弱(v1)**:声明制 + SDK 审计;无物理拦截 | 见 §3 |

结论:v1 的安全承诺是"**插件拿不走你的凭据、碰不了你的服务器(除非
授权)、搞不垮你的主程序**";"恶意插件不能读你本地磁盘"在 v1 **不成
立**,靠来源信任(签名分档,见 10)把关。此表在用户文档与插件安装
确认页如实呈现。

## 3. OS 级沙箱路线(v2+,预研项)

目标:把 §1 的两个"弱"升为"中/强",让 fs.local/network 权限获得物理
强制。按平台可行性排序:

| 平台 | 手段 | 预判 |
| --- | --- | --- |
| Linux | Landlock(文件范围)+ seccomp(syscall 面收敛)+ cgroups v2(资源) | 最可行,内核 ≥5.13 |
| macOS | `sandbox_init` profile(Apple 半私有但 Chromium 等在用)或 App Sandbox 化的 helper | 可行,需打磨 |
| Windows | Restricted token + Job Object 起步;AppContainer 为终态(管道 ACL 需配套放行) | 工作量最大 |

原则:沙箱是**能力收敛器**而非权限系统替代——Broker/授权 UX 不变,
沙箱把"合规契约"变成"物理不可为"。路线图上 M8 启动预研,先 Linux。
