# 01 · 愿景与目标

## 1. 愿景

VelaShell 今天是一个 SSH/终端/SFTP 工作台。插件系统的目标是把它变成一个
**平台**:核心保持精简、稳定、快,所有"锦上添花"的能力——AI 助手、媒体
查看、日志分析、批量运维自动化——都由插件提供,并且:

1. **隔离**:插件运行在独立进程,卡死或崩溃不影响主程序;插件之间互不可见。
2. **受控**:插件能做什么由用户说了算——Android 式权限模型,敏感能力
   首次使用时显式弹窗授权,可随时撤销。
3. **开放**:一套稳定的公开 SDK + 标准打包格式,任何人都能开发与发布插件。
4. **动态**:安装、启用、禁用、卸载、升级全部无需重启主程序。

## 2. 典型场景(驱动设计的用例)

设计中的每一项能力都必须能支撑至少一个下述场景;反过来,每个场景在
[14-roadmap.md](14-roadmap.md) 中都对应一个验收用的示例插件。

| # | 场景 | 需要的能力 |
| --- | --- | --- |
| S1 | **图片查看器**:在 SFTP 面板中双击远程图片,插件在文档区打开预览页,支持缩放 | 远程文件读取、文档页贡献点、图像表面(共享内存位图) |
| S2 | **MP3 播放器**:浏览本地/远程音乐文件并播放,状态栏显示播放进度 | 本地/远程文件读取、音频输出服务、VelaUI 面板、状态栏贡献点 |
| S3 | **AI 助手**:解释终端里选中的报错、根据自然语言生成命令并回填到终端 | 终端读取(选区/输出)、终端写入、AI 网关、侧栏视图 |
| S4 | **日志高亮/分析**:订阅终端输出流,识别错误模式并在侧栏聚合展示 | 终端输出订阅、侧栏视图、通知 |
| S5 | **自动化**:会话连接成功后自动执行一组命令;文件传输完成后触发远端校验 | 事件订阅、远程执行、自动化规则引擎 |
| S6 | **服务器仪表盘**:周期性在会话上执行采集命令,渲染 CPU/内存/磁盘图表 | 远程执行、定时触发器、VelaUI |
| S7 | **配置文件编辑增强**:为远程编辑场景提供额外语法定义或 lint | 远程文件读写、(远期)编辑器扩展点 |

## 3. 目标(Goals)

- G1:插件与主进程**进程级隔离**;单插件 CPU 100%、内存暴涨、崩溃、死锁均
  不影响主程序 UI 响应,且用户可从插件管理页一键终止。
- G2:插件之间默认**互相隔离**(独立进程、独立数据目录、不可互相调用)。
- G3:**权限系统**覆盖所有敏感能力;未声明的权限调用直接失败,危险权限
  首次使用弹窗,授权决定可持久化并可在设置页撤销。
- G4:**动态装卸**:安装/卸载/升级/禁用即时生效,不重启主程序;卸载后
  插件的贡献点、进程、临时资源全部清理。
- G5:**公开 SDK**:`VelaShell.PluginSdk` NuGet 包 + `dotnet new` 模板 +
  打包 CLI + F5 调试体验;SDK 面向 apiLevel 承诺向后兼容。
- G6:**远程能力**:插件可经授权访问用户已连接会话的 SFTP 与命令执行,
  复用宿主的连接(不重复认证、不另建连接,除非显式申请)。
- G7:**UI 扩展**:插件可以贡献命令、菜单、状态栏、侧栏视图、文档页、
  设置页,并通过 VelaUI 声明式协议渲染自定义界面。
- G8:**五语言一致性**:插件清单与贡献点文案支持本地化(`%key%` +
  `plugin.nls.<locale>.json`),与主程序现有五语言机制协同。
- G9:性能:主程序冷启动时间不因"已安装 N 个插件"显著劣化——插件默认
  惰性激活(激活事件驱动),启动期不拉起任何插件进程。

## 4. 非目标(Non-Goals)

- N1:**不做浏览器级安全沙箱(初期)**。权限系统的第一目标是*透明与用户
  控制*,不是对抗蓄意恶意的原生代码——插件进程仍是一个普通 OS 进程,
  恶意插件可绕过 Broker 直接调 OS API。对抗恶意代码依赖:签名与来源
  信任(见 10)、威胁模型中规划的 OS 级沙箱演进(见 12)。VSCode 的
  扩展宿主同样不沙箱扩展,这是行业常态,但我们必须在文档与 UI 中
  诚实告知用户。
- N2:**不支持进程内插件**。不提供"为了性能把插件加载进主进程"的后门,
  避免隔离承诺被侵蚀。宿主自身的内置功能不走插件通道。
- N3:**初期不做 WebView 型插件 UI**。Avalonia 生态内嵌 WebView 尚不
  成熟且体积代价大;UI 扩展以贡献点 + VelaUI 为主,列为远期再评估项。
- N4:**初期不做跨插件依赖/互调**(plugin A 依赖 plugin B)。留好
  manifest 字段但第一版拒绝安装带依赖的包。
- N5:**不做插件内嵌原生 UI 窗口**(插件自开窗口不受宿主管理,破坏
  体验一致性;技术上无法禁止,但不提供 API 支持且视为不合规行为)。
- N6:**非 .NET 插件(JS/Python/native)不在第一版范围**。IPC 协议本身
  语言中立(JSON-RPC),为远期多语言 SDK 留下空间,但第一版只发布
  .NET SDK。

## 5. 与现有系统的对比与借鉴

| 系统 | 借鉴 | 不采用 |
| --- | --- | --- |
| **VSCode** | 激活事件、贡献点声明、`package.json` 清单、扩展宿主进程、NLS 本地化机制 | 所有扩展共享一个宿主进程(一个扩展卡死拖累全部)——我们默认**每插件一进程**;webview UI |
| **Android** | 权限分级(normal/dangerous)、首次使用弹窗、设置页集中管理与撤销、权限组 | 安装时一次性授权全部(旧模型);我们不做 install-time 授权 |
| **Chrome 扩展** | 权限在商店页醒目展示、host permission(按目标域名/路径收窄授权范围) | — |
| **JetBrains 插件** | 兼容版本区间校验、市场审核流程 | 进程内加载(其插件崩溃可拖垮 IDE) |
| **OBS/Audacity 等原生插件** | — | 进程内原生插件模型(与隔离目标冲突) |

关键差异化决策:**默认每插件一个独立 PluginHost 进程**。VSCode 的单
扩展宿主是我们明确要避免的痛点(一个扩展死循环,所有扩展全部失灵)。
代价是每进程 ~30–60MB 的基础内存与进程启动时间,由"惰性激活 + 空闲
回收"(见 04)对冲。

## 6. 成功度量

- 杀死任一插件进程,主程序零感知(除该插件 UI 变灰),自动化测试覆盖。
- 插件从触发激活事件到 `Activate()` 返回 P95 < 800ms(暖机后 < 300ms)。
- 一个"Hello World"插件从 `dotnet new` 到在 VelaShell 里跑起来 ≤ 5 分钟。
- 三个官方示例插件(图片查看器、MP3 播放器、自动化)全部只用公开 SDK 实现。
