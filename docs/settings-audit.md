# VelaShell 设置界面功能审计与重组清单

> 审计范围:设置模型、设置页面、持久化服务及主要运行时消费者  
> 审计目标:识别功能重复、语义不清、运行时冲突、无效设置、缺失设置,并给出可执行的分组方案  
> 状态说明:本文是整改台账。2026-07-11 完成第一批整改(阶段一~三及条件展示),各表"状态"列已同步;文中源码行号为审计时快照,整改后可能偏移。

## 1. 执行摘要

当前设置中心包含 9 个一级入口,约 80 个可见设置或管理功能。主要问题不是设置数量本身,而是:

1. `BellMode` 与 `VisualBell` 同时控制终端响铃,存在真实运行时冲突。(已修复)
2. 自动重连次数、默认值和"显示隐藏文件"等设置存在多个状态来源。(已修复)
3. 导出、语言切换和关闭确认等功能的界面描述与实际行为不完全一致。(除本地化外已修复)
4. 12 个以上尚未实现的设置以禁用控件形式长期占据界面。(已全部隐藏或删除)
5. "常规"页承担启动、连接、通知、安全、日志、更新等多个领域,已经成为杂项集合。(部分缓解:更新组隐藏、重连设置归组;整体 IA 重组待做)
6. 密钥管理和代码片段属于功能管理工具,不属于应用偏好设置。(待做)
7. 终端颜色位于"外观",终端字体、编码、光标等位于"终端",用户需要跨页寻找同一对象的设置。(待做)

### 1.1 优先级定义

| 优先级 | 含义 | 处理要求 |
|---|---|---|
| P0 | 已确认的安全风险或运行时冲突 | 应优先处理,避免继续产生错误配置 |
| P1 | 明显误导、状态不一致或关键体验缺陷 | 建议进入最近一个整改版本 |
| P2 | 信息架构、可发现性和高级体验问题 | 可在设置界面重组时统一处理 |
| P3 | 可选增强 | 根据用户反馈和产品规划决定 |

### 7.1 不建议暴露为普通用户设置的内部参数

| 参数 | 原因 |
|---|---|
| SSH 流读取缓冲区 | 属于吞吐与延迟的内部调优参数,用户难以判断正确值 |
| ShellStream 内部缓冲区 | 应由实现根据连接与平台调优 |
| Bridge dispose 等待时间 | 生命周期实现细节 |
| Echo 抑制次数和时间窗口 | 协议桥接实现细节 |
| 网络活动箭头字节阈值 | 视觉实现细节,暂无足够用户价值 |

### 2026-07-12 第二批(主机指纹确认三选项 + 已信任主机管理)

背景:用户开启“首次连接需人工确认指纹”后未见弹窗——原因不是链路缺失,而是目标主机早已在开关开启前经 TOFU 自动记录(持久化在 SonnetDB `known_hosts` 集合,不再是 `~/.velashell/known_hosts.json`,该文件仅作首次运行的一次性导入源),且没有任何 UI 可删除已信任记录。

- **确认弹窗改为三选项**(与主流 SSH 客户端一致):永久信任(写入 known_hosts)/ 仅本次信任(进程内 `HostTrustOnceCache`,不落盘,重启后再次询问)/ 取消(拒绝并中止连接,fail-closed)。`IHostKeyPrompt.ConfirmAsync(bool)` 重构为 `DecideAsync(HostKeyDecision)`;指纹变更(关闭阻断开关时)走同一三选项。
- **新增“已信任主机”管理**(设置 → 安全审计):列出 host:port、密钥类型与 SHA256 指纹,可逐条删除;删除即时生效,下次连接重新执行首次确认流程。
- **修复 SFTP 通道不校验主机指纹的缺口**:此前 SFTP 独立通道未订阅 `HostKeyReceived`(= 默认信任任意指纹,存在 MITM 风险);现严格比对 known_hosts 或本次运行的临时信任,不匹配即拒绝、不弹窗。
- 安全告警新增 `hostkey-trusted-once` 事件类型。

**验证**:全量构建通过;VelaShell.Tests 268 项、Core.Tests 148 项通过(弹窗三选项新增用例)。
